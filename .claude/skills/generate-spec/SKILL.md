# planning/ → SPEC.md 자동 생성 워크플로우

> `planning/` 폴더의 기획 문서를 분석하여 `SPEC.md`를 자동 생성하는 워크플로우.
> 사용자가 "planning 폴더를 읽고 SPEC.md를 작성해줘"라고 요청하면 이 절차를 따릅니다.

---

## Step 1: planning/ 폴더 스캔

`planning/` 폴더에서 지원 확장자 파일을 모두 찾습니다.

**지원 확장자**: `.md`, `.txt`, `.pdf`

```bash
# 파일 목록 확인
ls planning/
```

- 파일이 없으면 사용자에게 안내: "planning/ 폴더에 기획 문서를 넣어주세요. (md, txt, pdf 지원)"
- PDF 파일은 Read 도구로 읽기 (페이지 수가 많으면 pages 파라미터 사용)

## Step 2: 모든 파일 읽기

발견된 모든 기획 문서를 읽습니다.

- `.md`, `.txt` → Read 도구로 읽기
- `.pdf` → Read 도구로 읽기 (10페이지 초과 시 분할 읽기)
- 여러 파일이 있으면 병렬로 읽기

## Step 3: SPEC.template.md 참조

`SPEC.template.md` 파일을 읽어 10개 섹션 구조를 참조합니다.

**SPEC.md 10개 섹션:**

| # | 섹션 | 필수 | 핵심 내용 |
|---|------|------|----------|
| 1 | 프로젝트 기본정보 | 필수 | 이름, slug, 설명 |
| 2 | 사용자 역할 | 필수 | 역할 정의 (admin 필수) |
| 3 | 데이터 모델 | 필수 | 테이블, 필드, FK 관계 |
| 4 | API 엔드포인트 | 필수 | REST API, crud 키워드 |
| 5 | 페이지 구성 | 필수 | 프론트엔드 페이지, 구성 요소 |
| 6 | 레이아웃 | 필수 | 사이드바 메뉴, 아이콘, 역할 제한 |
| 7 | 추가 기능 | 선택 | 체크리스트 |
| 8 | 배포 설정 | 선택 | 도메인, DB |
| 9 | 에이전트 설정 | 선택 | AI/챗봇 관련 시에만 |
| 10 | 디자인 요구사항 | 선택 | 브랜드 컬러 등 언급 시에만 |

## Step 4: 기획 문서 분석 → SPEC 섹션 매핑

기획 문서 내용을 분석하여 각 섹션에 매핑합니다.

### 섹션 1: 프로젝트 기본정보

- 프로젝트 이름, 한 줄 설명 추출
- slug: 프로젝트 이름에서 kebab-case 변환 (한국어면 영문 slug 생성)
- 도메인: 기획 문서에 언급 없으면 `<!-- TODO: 작성 필요 -->`

### 섹션 2: 사용자 역할

- 기획 문서에서 사용자 유형/역할 식별
- `admin` 역할은 **반드시 포함** (기획 문서에 없어도 추가)
- 승인 워크플로우 언급 시 `pending` 역할 추가
- 역할별 권한 수준 정의

### 섹션 3: 데이터 모델

- 기획 문서에서 엔티티(리소스) 식별
- 각 엔티티의 필드를 SPEC.template.md 타입으로 변환:
  - 문자열 → `String(N)` (제목: 200, 이름: 100, 짧은 텍스트: 50)
  - 긴 텍스트 → `Text`
  - 숫자 → `Integer` 또는 `Float`
  - 날짜 → `DateTime`
  - 참/거짓 → `Boolean`
  - 선택지 → `Enum(값1, 값2, ...)`
  - JSON 데이터 → `JSON`
  - 관계 → `FK → 모델명`
- 제약조건 설정: `NOT NULL`, `nullable`, `unique`, `default=값`, `CASCADE`
- BaseModel(id, created_at, updated_at)은 자동 포함이므로 **생략**

### 섹션 4: API 엔드포인트

- 섹션 3에서 정의한 각 모델에 대해 API 설계
- CRUD 기본 동작이면 `crud` 키워드 사용
- 추가 엔드포인트(상태 변경, 검색 등)는 별도 테이블에 추가
- 권한은 섹션 2의 역할과 일치하게 설정
- 검색/필터 파라미터 명시: `?search=키워드&status=값&page=1`

### 섹션 5: 페이지 구성

- 기획 문서에서 화면/페이지 식별
- 각 페이지의 경로, 레이아웃(public/authenticated), 인증 여부 설정
- 구성 요소를 **구체적으로** 기술 (테이블 컬럼, 버튼명, 필터 등)
- 로그인/회원가입 페이지는 기본 포함

### 섹션 6: 레이아웃 (사이드바)

- 섹션 5의 주요 페이지를 사이드바 메뉴로 구성
- 아이콘은 lucide-react에서 적절한 아이콘 선택
- 역할 제한 설정 (관리자 전용 메뉴 등)

### 섹션 7: 추가 기능

- 기획 문서에서 언급된 추가 기능 체크리스트로 변환:
  - 이메일 인증, 승인 워크플로우, 파일 업로드, 실시간 알림 등
- 언급되지 않은 항목은 `[ ]` (미체크)로 유지

### 섹션 8: 배포 설정

- 배포 관련 언급이 있으면 반영
- 없으면 기본값(Railway, PostgreSQL) 사용 + 도메인 `<!-- TODO: 작성 필요 -->`

### 섹션 9: 에이전트 설정 (조건부)

**기획 문서에 AI, 챗봇, 에이전트, 자동 분석 등이 언급된 경우에만 생성.**

- 메인 에이전트: 이름, 시스템 프롬프트
- 서브에이전트: 역할 분리가 필요한 경우
- 커스텀 도구: DB 조회 도구 (섹션 3 모델 기반)
- RAG 소스: 문서 검색이 필요한 경우
- **언급 없으면 이 섹션 전체를 생략**

### 섹션 10: 디자인 요구사항 (조건부)

**기획 문서에 브랜드 컬러, 디자인 가이드, UI 스타일 등이 언급된 경우에만 생성.**

- 브랜드 컬러를 Tailwind 클래스로 변환
- 레이아웃 원칙, 컴포넌트 스타일 명세
- **언급 없으면 이 섹션 전체를 생략** (기본 디자인 토큰 사용)

## Step 5: SPEC.md 파일 생성

분석 결과를 `SPEC.template.md` 구조에 맞춰 `SPEC.md`를 생성합니다.

### 핵심 규칙

1. **SPEC.template.md의 마크다운 구조를 정확히 따름** — 테이블 형식, 섹션 제목, 주석 스타일
2. **기획 문서에 없는 정보는 추측하지 않음** — `<!-- TODO: 작성 필요 -->` 주석으로 표시
3. **데이터 모델 타입은 가이드 준수** — `String(N)`, `Text`, `Enum(값1, 값2)`, `FK → 모델명` 등
4. **admin 역할은 반드시 포함** — 기획 문서에 없어도 자동 추가
5. **섹션 9, 10은 관련 내용이 있을 때만 포함** — 없으면 섹션 자체를 생략
6. **한국어 사용** — UI 텍스트, 설명, 버튼명 모두 한국어
7. **SPEC.template.md의 주석(<!-- -->)은 제거** — 실제 값으로 교체된 상태로 생성

### 생성 위치

```
SPEC.md  ← 프로젝트 루트에 생성
```

## Step 6: 검토 요청

생성 완료 후 사용자에게 다음 정보를 안내합니다:

### 1) 생성 완료 요약

```
SPEC.md 생성 완료!

기획 문서에서 추출한 내용:
- 프로젝트: {프로젝트 이름}
- 역할: {역할 수} 개
- 데이터 모델: {모델 수} 개
- API 엔드포인트: {엔드포인트 수} 개
- 페이지: {페이지 수} 개
- 에이전트: {있음/없음}
- 디자인 커스텀: {있음/없음}
```

### 2) 누락/모호한 부분 안내

`<!-- TODO: 작성 필요 -->`로 표시된 항목을 목록으로 정리:

```
검토가 필요한 항목:
- [ ] 섹션 1: slug 확인 필요
- [ ] 섹션 3: {모델명}의 {필드명} 타입/제약조건 확인 필요
- [ ] 섹션 8: 배포 도메인 작성 필요
```

### 3) 다음 단계 안내

```
SPEC.md를 검토한 뒤, "SPEC.md를 읽고 프로젝트를 생성해줘"라고 요청하면
14단계 자동 생성 워크플로우가 시작됩니다.
```

---

## 주의사항

- `planning/` 폴더가 없거나 비어있으면 사용자에게 안내
- 기획 문서가 여러 개일 경우 모든 문서를 종합하여 하나의 SPEC.md 생성
- 기존 SPEC.md가 있으면 덮어쓰기 전에 사용자 확인 요청
- PDF 파일이 20페이지 초과일 경우 분할하여 읽기
